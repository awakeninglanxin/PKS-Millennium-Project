# 刀二：MoE 路由的数学闭环 — 从梯度训练到二分图最优匹配

> **动什么**: MoE 的 load-balancing aux-loss (2017 Shazeer heuristics) → 闭式最优路由  
> **复杂度收益**: 消掉 aux-loss 超参 + 专家分配从"软搜"变"硬算"  
> **风险**: 🟡 中等 — K3 Quantile Balancing 已半步, 闭式化是自然延伸

---

## 一、MoE 路由的三代进化

```
2017 Shazeer:       router(x) = softmax(Wx), Top-K 选择, aux-loss 约束负载
                    → aux-loss 系数是敏感超参, 训崩常见

2024 DeepSeek-V3:   每个专家挂 bias term, Top-K 选择时加到亲和度
                    → bias 动态调 (负载高减、低加), 主 loss 不动
                    → aux-loss 去了, 但 bias 调整步长 γ 仍是超参

2026 K3 Quantile:   按 router-score 分位数直接派专家
                    → 消掉 bias 调整步长, 完全无训练参数
                    → 分位数切点 = 唯一的自由度

下一刀 闭式匹配:    把 Top-K 选择替换为最小费用最大流匈牙利算法
                    → 全局最优的专家分配, 无需任何超参
```

## 二、为什么路由可以走闭式解

### 2.1 MoE 路由本质是二分图匹配

把每一层的 expert 选择看作一个**二分图**:

```
左边: T 个 token (每个 token 需要分配给 1 个专家)
右边: E 个 expert (每个 expert 有容量上限)
边权重: router_score(token, expert)
目标: 总权重最大的合法匹配
```

这正是**带容量约束的最大权二分图匹配**——运筹学里已经解决了几十年的经典问题。

### 2.2 匈牙利算法 → O(TE√(T+E))

```
标准匈牙利算法: O(V³) 严格匹配
加速版 (Hopcroft-Karp + 费用流): O(TE√(T+E)) 带容量约束

K3 场景: T=序列长度, E=896, 激活 16 个/每 token
         → 等效为每 token 从 896 专家中选 16 个成本最小的 → 费用流
         → O(T × 896 × √(T+896)) ≈ O(T√T) in T, O(E) in E
```

对于 K3 的 MoE blocking: 一个 MoE block 里有 896 个专家，每个 token 激活 16 个。**费用流算法可以在每次前向传播中，为所有 token 计算出全局负载均衡的最优专家分配**——不需要任何梯度、不需要任何 aux-loss、不需要任何超参。

### 2.3 为什么不直接用？

因为匈牙利算法在 GPU 上跑不过 softmax+top-k 的矩阵乘法。但 MoE 层是**隔几层才有一次**（K3 约 1/3 层是 MoE），所以开销可接受。而且 2025 年已经有 GPU 上的高效匈牙利实现（cuGraph 等）。

**折中方案**: 训练时用 Quantile Balancing (K3 当前方法) → 推理/微调时切闭式匹配 → 差异极小但去掉所有启发式。

## 三、与 K3 Quantile Balancing 的衔接

```
当前 K3 Quantile Balancing:
  router_score = softmax(W_gate · x)
  quantile_thresholds = [0, 1/16, 2/16, ..., 15/16] × range(scores)
  每个 token 落到哪个分位 → 分配给对应专家

升级闭式匹配:
  step 1: 仍用 router_score (不改变训练好的权重)
  step 2: 把 Top-16 softmax 选择 → 替换为费用流最优化
  step 3: 容量约束 = 每个 expert 最多 batch/E 个 token
  step 4: 最小化总费用 = 最大化 router_score 之和
```

**关键**: 不改变训练的 router 权重——只是把 Top-K 贪心选择换成全局最优匹配。理论上应该比贪心更好（贪心可能让某些 expert 超载），且不需要重新训练。

## 四、为什么这对 K3 特别有意义

K3 的 896 专家/16 激活 (稀疏度 56) 已经是当前开源模型的极限。再往上加专家，routing 的脆弱性会指数级增长:

| 专家数 | Top-K | 每个 token 的选择空间 | routing 脆弱性 |
|:--:|:--:|:--:|:--:|
| 8 | 2 | C(8,2)=28 | 低 |
| 384 (K2) | 8 | C(384,8)≈10^16 | 中 |
| 896 (K3) | 16 | C(896,16)≈10^32 | 高 |
| 2048 (K4 推测) | 32 | 天文 | **极高** |

896 专家时，贪心 Top-K 的局部最优和全局最优之间的差距已经开始显著。闭式匹配恰好在这个规模上开始有真正的收益——**不是锦上添花，是让 896→2048 的 scale 路径不被 routing 瓶颈卡死**。

## 五、与 PKS 方法论的关系

这是 **"算数学 > 学数学"** 的又一个完美案例:

- aux-loss 是**学**（通过梯度下降学"什么是好的专家分配"）
- 闭式匹配是**算**（二分图的最优匹配有确定性的算法）
- 我们不需要"学出一个更好的 aux-loss"——我们直接算出来最优分配

和 FFT 替代时域卷积、Hungarian 替代 aux-loss——**同一个 DNA**。
